## Why Cost Optimization Matters

When I started consulting for enterprise clients on Azure, the first surprise was always the monthly bill. Most teams celebrate launching to production, then face sticker shock at the end of the month.

The good news: most Azure environments have **30-50% waste** that can be eliminated without affecting performance or reliability.

Here are five strategies I apply to every cloud assessment.

## 1. Right-Sizing Virtual Machines

Most VMs are provisioned with buffer for "growth that never comes". Azure Advisor identifies underutilized VMs based on 14-day CPU and memory metrics.

```bash
az advisor recommendation list \
  --category Cost \
  --query "[?impactedField=='Microsoft.Compute/virtualMachines']" \
  --output table
```

Typical savings: **20-30% per VM** by moving from D-series to B-series for non-production workloads.

## 2. Reserved Instances and Savings Plans

For predictable workloads running 24/7, Reserved Instances (RIs) save up to **72%** compared to pay-as-you-go.

Decision matrix:

| Workload Type | Recommendation |
|---|---|
| Production, predictable | 3-year Reserved Instance |
| Variable but consistent | 1-year Savings Plan |
| Dev/Test | Auto-shutdown + Spot VMs |

## 3. Auto-Shutdown for Dev/Test

The simplest cost killer. Most dev environments run 168 hours per week but are used 40.

In Azure, enable auto-shutdown on every dev VM:

```bash
az vm auto-shutdown \
  --resource-group dev-rg \
  --name dev-vm-01 \
  --time 1900
```

This single change typically saves **70% on dev costs**.

## 4. Storage Tier Optimization

Azure Blob Storage has three tiers: Hot, Cool, Archive. Most teams default to Hot, but logs and backups belong in Cool or Archive.

Lifecycle policy example: move blobs to Cool after 30 days, Archive after 90 days. Savings: **80% on storage costs**.

## 5. Tagging and Showback

You cannot optimize what you cannot measure. Implement a tagging strategy:

- `Environment`: prod, staging, dev
- `Owner`: team name
- `CostCenter`: department code
- `Project`: project identifier

Then build cost dashboards in Azure Cost Management showing spend per team. When teams see their costs, behavior changes overnight.

## Closing Thoughts

Cost optimization is not a one-time project. It is a continuous practice that should be embedded in your engineering culture. Start with the quick wins above, then build the discipline.

If you are starting your FinOps journey, focus on **visibility first, optimization second**. You cannot fix what you cannot see.
